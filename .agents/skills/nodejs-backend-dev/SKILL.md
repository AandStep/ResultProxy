# Node.js Backend Developer Skill

## Обзор
Данный скилл определяет стандарты разработки бэкенда для обеспечения высокой производительности, безопасности и чистоты архитектуры. Фокус на разделении ответственности, надежной обработке асинхронных операций и корректной интеграции с Electron IPC.

## Когда использовать
- При создании новых API эндпоинтов (Express или IPC).
- При разработке модулей бизнес-логики (Services).
- При взаимодействии с файловой системой или системными ресурсами ОС.
- При реализации долгоживущих процессов или периодических задач.

## Архитектурные паттерны

### 1. Layers & Dependency Injection (DI)
Соблюдайте четкое разделение:
- **System/Hardware**: Прямое взаимодействие с ОС (System Factory/Adapters).
- **Domain Service**: Бизнес-логика и оркестрация данных.
- **Controller/Handler**: Точки входа (Express routes, IPC handlers).

**Правило:** Экземпляры сервисов создаются и связываются централизованно (обычно в `electron-main.cjs`). Избегайте "скрытых" зависимостей через глобальные `require` внутри сервисов, если это инстансы других сервисов.

```javascript
// ✅ Правильно: Передача зависимостей через конструктор
class ProxyManager {
  constructor(logger, systemAdapter, stateStore) {
    this.logger = logger;
    this.system = systemAdapter;
    this.state = stateStore;
  }
}
```

### 2. Logging & Error Handling
- **НИКОГДА** не используйте `console.log`. Используйте `loggerService`.
- Оборачивайте асинхронные операции в `try-catch`.
- Всегда логируйте контекст ошибки и уровень (info, warning, error).

### 3. State Management
Для хранения и синхронизации состояния между процессом Node.js и UI используйте `stateStore`.
- Все изменения состояния должны проходить через `stateStore.update()`.
- Чтение состояния — `stateStore.getState()`.

## Electron IPC Security
- Используйте `ipcMain.handle` и `ipcRenderer.invoke` для асинхронных запросов-ответов.
- Не передавайте чувствительные объекты целиком; возвращайте только необходимые данные.
- Обязательно валидируйте входящие аргументы в `ipcMain`.

## Реализация (Пример сервиса)

```javascript
// backend/services/system.service.cjs
class SystemService {
  constructor(logger) {
    this.logger = logger;
  }

  async getStats() {
    try {
      this.logger.log('Запрос статистики системы', 'info');
      // ... логика получения данных
      return { status: 'ok' };
    } catch (error) {
      this.logger.log(`Ошибка статистики: ${error.message}`, 'error');
      throw error;
    }
  }
}
```

## Распространенные ошибки
- **Sync block**: Использование синхронных версий функций `fs` или `child_process` (кроме инициализации).
- **Circular dependencies**: Неправильный импорт модулей, приводящий к зацикливанию.
- **Global objects**: Использование глобальных переменных для хранения состояния вместо `stateStore`.

## Таблица рационализаций (Rationalization Table)

| Оправдание | Реальность |
|------------|------------|
| "Это просто лог для отладки, потом удалю" | Временные логи часто остаются в коде. Используйте `loggerService` сразу. |
| "Слишком простой метод для DI" | Простота обманчива. DI облегчает тестирование и мокирование. |
| "Я обновлю состояние напрямую, это быстрее" | Прямое обновление ломает синхронизацию с UI и другими слоями. Используйте `stateStore`. |
| "Try-catch здесь не нужен, ошибка исключена" | В Node.js (особенно с IO/ОС) ошибки возможны всегда. Оберните асинхронный вызов. |

## Список признаков (Red Flags) - ОСТАНОВИТЕСЬ И ПЕРЕДЕЛАЙТЕ
- Использование `console.log`, `console.warn`, `console.error`.
- Отсутствие `try-catch` в асинхронных методах сервисов.
- Прямой `require` соседних сервисов-синглтонов (используйте DI через конструктор).
- Прямая мутация объектов в `stateStore` без использования `.update()`.
- Использование синхронных методов `fs` (`readFileSync`, etc.) вне этапа старта приложения.
